# 短视频智能切片与素材管理系统 — 设计文档 v2.0

> 版本：v3.0
> 日期：2026-03-09
> 基于：v2.0 设计文档 + 落地风险审查
> 变更摘要：
> - **[安装风险]** 人脸识别库从 `face_recognition(dlib)` 换为 `InsightFace`（pip直装，M系列Mac原生支持，精度更高）
> - **[识别准确率]** LLM兜底从"发文字描述"改为"直接发裁剪帧图片 + 候选列表"，充分利用Claude视觉能力
> - **[SKU混淆]** 新增CLIP二阶段细粒度对比 + 候选图LLM目视判断，解决同品牌相似SKU混淆
> - **[纠错体验]** 触发阈值从"≥10条"改为"≥3条快速触发 + 每日批跑"，消除前9条纠正无效的体验断层
> - **[用户体验]** 新增M14本地预览服务（FastAPI，`vidslice serve`一行启动），补上"不需要懂技术"的承诺

---

## 一、项目概述

### 1.1 一句话定义

把不可复用的营销短视频，转化为**可搜索、可组合、可复利的素材资产**。

### 1.2 版本规划

| 版本 | 范围 | 交互方式 | 目标 |
|------|------|---------|------|
| **V1（本文档）** | 切片 + 识别 + 归档 + 浏览/纠错 | CLI + 飞书 | 建立素材资产库，跑通完整闭环 |
| V2 | + 智能组合生成视频 | + Web 管理后台 | 素材复利，自动生成新视频 |
| V3 | + 效果追踪 + 数据驱动优化 | + 数据看板 | 闭环优化 |

### 1.3 V1 核心用户故事

```
作为运营人员，我希望：
1. 上传一批营销视频后，系统自动切片并识别出每个片段里是谁、是什么产品
2. 我能浏览、搜索、筛选这些片段
3. 识别错误时我能一键纠正，系统会越来越准
4. 新增 KOL 或产品时，我能简单地录入参考图
5. 整个过程不需要我懂技术
```

### 2.2 三层处理架构（保留 v1.0 设计）

| 层级 | 处理内容 | 工具 | 成本 | 调用比例 |
|------|---------|------|------|---------|
| L1 | 检测 + 初步分类 | YOLO / MediaPipe | 免费 | 100% |
| L2 | 特征提取 + 身份识别 | InsightFace + CLIP + FAISS | 免费 | 100% |
| L3 | LLM 兜底复核 | Claude Haiku（直接发送帧图片） | 按量付费 | <5% |

### 2.3 设计原则（v2.0 新增）

| 原则 | 说明 | 对应 v1.0 问题 |
|------|------|---------------|
| **配置优于代码** | 所有业务规则、阈值、分类从配置文件读取 | 23 处硬编码 |
| **接口优于实现** | 识别器、存储、通知均面向接口 | 紧耦合 |
| **组合优于枚举** | 标签 + 权重代替固定分类 | 分类硬编码 3 种 |
| **用户旅程完整** | 冷启动、浏览、纠错、搜索均有设计 | v1.0 全部缺失 |

## 三、配置中心设计

> 解决 v1.0 复盘中识别的 23 处硬编码问题

### 3.1 配置文件结构

```
config/
├── system.yaml              # 系统级配置（阈值、重试策略等）
├── brands/                  # 按品牌组织
│   ├── tulandut/
│   │   ├── brand.yaml       # 品牌基础信息
│   │   ├── recognizers.yaml # 该品牌的识别目标
│   │   └── categories.yaml  # 该品牌的分类规则
│   └── brand_b/
│       └── ...
├── models.yaml              # LLM / CV 模型路由配置
└── notifications.yaml       # 通知渠道配置
```

### 3.2 system.yaml — 系统级配置

```yaml
# 帧抽取
frame_extraction:
  default_fps: 2              # 每秒抽帧数，可按视频类型覆盖
  strategies:
    fast_scan: { fps: 1 }
    detailed: { fps: 5 }

# 识别阈值
recognition:
  face_confidence_threshold: 0.85    # 人脸识别置信度阈值
  product_top_k: 5                   # 产品匹配返回前 K 个
  product_confidence_threshold: 0.80
  llm_fallback_threshold: 0.60      # 低于此值触发 LLM 兜底

# 向量库
vector_store:
  engine: "faiss"            # faiss | milvus | pinecone
  face_dimensions: 128       # face_recognition 默认维度
  product_dimensions: 512    # CLIP 默认维度

# 切片
clipping:
  min_duration: 1.0          # 最短片段（秒）
  max_duration: 30.0         # 最长片段（秒）
  merge_gap: 0.5             # 两个同类片段间隔小于此值则合并

# 重试策略
retry:
  default:
    max_attempts: 3
    strategy: "exponential_backoff"
    base_delay: 1.0

# 存储
storage:
  engine: "local"            # local | s3 | oss
  base_path: "./media_library"
  path_template: "{brand}/{category}/{person_or_product}/{date}_{clip_id}.mp4"
```

### 3.3 brands/{brand}/recognizers.yaml — 识别目标配置

```yaml
# 泛化实体识别器（解决 v1.0 问题 #4：识别目标固定为人和产品）
entity_types:
  - type: "person"
    engine: "face_recognition"
    vector_store: "faces.index"
    threshold: 0.85
    enabled: true

  - type: "product"
    engine: "clip_faiss"
    vector_store: "products.index"
    threshold: 0.80
    enabled: true
    # v3.0 新增：同品牌SKU混淆处理
    disambiguation:
      enabled: true
      # 当top-1和top-2置信度差值 < margin 时，触发二阶段细粒度对比
      margin: 0.08
      # 二阶段：裁剪ROI后用更高分辨率重新提取特征对比
      second_stage_engine: "clip_faiss_hires"  # ViT-L/14，更细粒度
      # 二阶段仍不确定 → 送LLM，提供top-K候选图片让模型目视判断
      llm_fallback_with_candidates: true

  # 可扩展：未来新增识别目标只需加配置，不改代码
  # - type: "scene"
  #   engine: "clip_faiss"
  #   vector_store: "scenes.index"
  #   threshold: 0.70
  #   enabled: false
```

### 3.4 brands/{brand}/categories.yaml — 分类规则配置

```yaml
# 标签 + 权重系统（解决 v1.0 问题 #1：分类硬编码 3 种）
classification:
  detected_elements: ["person", "product"]   # 当前启用的检测元素

  rules:
    - id: "person_only"
      label: "纯人物"
      conditions:
        - "person >= 0.7"
        - "product < 0.1"

    - id: "product_only"
      label: "纯产品"
      conditions:
        - "product >= 0.7"
        - "person < 0.1"

    - id: "person_and_product"
      label: "人物+产品"
      conditions:
        - "person >= 0.3"
        - "product >= 0.3"

    - id: "other"
      label: "其他"
      conditions:
        - "default"    # 兜底分类
```

### 3.5 models.yaml — 模型路由配置

```yaml
# 解决 v1.0 问题 #5：LLM 模型选择硬编码
model_routing:
  llm_fallback:
    primary:
```

### 4.2 冷启动流程（v1.0 缺失 — 新增）

#### 4.2.1 录入人物

```
用户操作（CLI）：
$ vidslice identity add --brand tulandut --type person --name "张三"
  --photos ./张三_正脸.jpg ./张三_侧脸.jpg ./张三_工作照.jpg

系统处理：
1. 检测每张图中的人脸，提取 128 维向量
2. 计算向量质量（正脸/侧脸覆盖度）
3. 存入 faces.index，绑定 person_id
4. 返回确认：
   ✅ 张三 已录入（3张参考图，向量质量：优）
   💡 建议：再补充 1 张戴眼镜/帽子的照片可提升识别率
```

**最低要求：** 每人 3 张参考图（正脸、左侧脸、右侧脸）
**推荐：** 5-8 张，覆盖不同光照、角度、装扮

#### 4.2.2 录入产品

```
用户操作（CLI）：
$ vidslice identity add --brand tulandut --type product --name "冰丝内衣A款"
  --sku "SKU_001"
  --photos ./产品A_正面.jpg ./产品A_侧面.jpg ./产品A_穿着效果.jpg

系统处理：
1. CLIP 提取每张图的 512 维特征向量
2. 存入 products.index，绑定 product_id
3. 返回确认：
   ✅ 冰丝内衣A款 (SKU_001) 已录入（3张参考图）
```

#### 4.5.2 纠正数据 → 向量库进化

```
纠正数据处理流程：

1. 用户纠正 → 存入 corrections 表
2. 立即更新该 clip 的元数据（实时生效）
3. 触发向量库更新的条件（满足任一即触发）：
   - 同一实体累计新增 ≥ 3 条纠正（快速响应）
   - 每日定时批跑（凌晨1点，处理当日所有纠正）
   - 手动触发：`vidslice retrain --brand tulandut --entity 李四`
4. 更新流程：
   a. 从纠正样本中提取新特征向量
   b. 追加到对应实体的向量集合（不覆盖原有向量）
   c. 重新评估匹配阈值（取新旧向量集合的最优分割点）
   d. 写入 recognizers.yaml
5. 飞书通知：
   "📈 识别模型已更新：李四 的识别准确率从 87% 提升到 93%（基于 3 条新纠正）"

> **v2.0改进说明**：原设计"≥10条才触发"导致前9条纠正完全不生效，体验割裂。改为3条快速触发+每日批跑兜底，兼顾响应速度和稳定性。
```

### 5.2 M03 实体识别器（v2.0 重构）

> 解决 v1.0 问题 #4：泛化为通用实体识别器，取代独立的 M03 人脸识别器 + M04 产品识别器

```
接口定义：

输入：
  - crop_image: 裁剪后的图像区域
  - entity_type: string           # 从 recognizers.yaml 读取
  - recognizer_config: dict       # 对应识别器的配置

输出：
  EntityMatch {
    entity_type: string           # "person" | "product" | 未来可扩展
    entity_id: string             # "张三" | "SKU_001"
    confidence: number            # 0.0-1.0
    source: "local" | "llm"       # 识别来源
    metadata: dict                # 附加信息（如 bbox 坐标）
  }

处理逻辑：
  1. 根据 entity_type 加载对应的识别引擎和向量库
  2. 执行匹配，返回 top-K 结果
  3. 最高置信度 ≥ threshold → 返回结果
  4. 最高置信度 < llm_fallback_threshold → 转 M04 LLM 兜底
  5. 中间地带 → 标记为 "uncertain"，返回但带警告
```

### 5.3 M08 身份录入服务（v2.0 新增）

```
接口定义：

CLI 命令：vidslice identity add / import / verify / list / remove

处理逻辑：
  add:
    1. 接收实体类型 + 名称 + 参考图片列表
    2. 按实体类型选择向量提取方式：
       - person → face_recognition 提取 128 维向量
       - product → CLIP 提取 512 维向量
    3. 质量检查：
       - person: 检测到人脸数量、角度覆盖度
       - product: 图片清晰度、背景复杂度
    4. 存入向量库 + 写入 identity 元数据表
    5. 返回录入结果 + 质量建议

  verify:
    1. 读取当前品牌下所有已录入实体
    2. 检查每个实体的向量数量和质量
    3. 输出健康报告 + 改进建议
```

### 5.4 M09 纠错处理器（v2.0 新增）

```
接口定义：

输入：
  Correction {
    clip_id: string
    field: "person_id" | "product_id" | "category"
    old_value: string
    new_value: string
    corrected_by: string          # 纠正人
    corrected_at: datetime
  }

处理逻辑：
  1. 更新 clip 元数据中的识别结果
  2. 存入 corrections 表
  3. 检查该实体的累计纠正数量：
     - < 10 条：仅记录
     - ≥ 10 条：触发向量库更新流程
       a. 从纠正样本中提取特征向量
       b. 追加到对应实体的向量集合
       c. 重新评估阈值
       d. 通知：识别准确率变化
  4. 纳入漂移检测统计
```

### 5.6 M11 任务调度器（v2.0 新增）

```
职责：
  - 管理视频处理的任务队列
  - 单视频处理：实时反馈进度
  - 批量处理：并发控制 + 进度汇总
  - 失败重试（策略从 system.yaml 读取）
  - 处理完成后触发 M12 通知

任务状态机：
  pending → processing → completed
                      → failed → retrying → completed
                                          → dead (超过最大重试次数)

并发控制：
  - 默认并发：3 个视频同时处理
  - CPU 密集阶段（L1/L2）：按 CPU 核数调整
  - LLM 调用阶段（L3）：受限于 API rate limit
```

## 八、处理流程详解

### 8.1 单视频处理 Pipeline

```
video.mp4
    │
    ▼
┌─────────────────────────────────────────────────┐
│ M01: 抽帧                                        │
│ · 读取 system.yaml → fps 配置                    │
│ · FFmpeg 按配置抽帧                              │
│ · 输出: frames/frame_001.jpg, frame_002.jpg ...  │
│ · 耗时: ~2秒/分钟视频                            │
└─────────────────────┬───────────────────────────┘
                      ▼
┌─────────────────────────────────────────────────┐
│ M02: 帧检测 + 分类                               │
│ · YOLOv8 检测每帧中的人/物体                     │
│ · 读取 categories.yaml → 匹配分类规则            │
│ · 连续同类帧合并为 segments                      │
│ · 输出: segments[] (每段的起止帧+分类+bbox)      │
│ · 耗时: ~50ms/帧                                 │
└─────────────────────┬───────────────────────────┘
                      ▼
┌─────────────────────────────────────────────────┐
│ M03: 实体识别                                    │
│ · 读取 recognizers.yaml → 加载启用的识别器       │
│ · 对每个 segment 的关键帧:                       │
│   - 有人脸 → face_recognition 匹配              │
│   - 有产品 → CLIP+FAISS 匹配                    │
│ · 置信度 ≥ threshold → 确认                      │
│ · 置信度 < fallback_threshold → 送 M04           │
│ · 输出: EntityMatch[] per segment                │
│ · 耗时: ~100ms/帧                                │
└───────────┬─────────────────────┬───────────────┘
            │                     │
     ≥ threshold           < fallback_threshold
            │                     │
            ▼                     ▼
┌────────────────┐    ┌─────────────────────────┐
│ 直接确认        │    │ M04: LLM 兜底            │
│                │    │ · 直接发送裁剪帧图片      │
│                │    │ · + 候选实体列表作为上下文│
│                │    │ · Claude Haiku 视觉判断   │
│                │    │ · 高置信结果 → 进入纠正池 │
│                │    │ · 耗时: ~3秒/帧           │
└───────┬────────┘    └───────────┬─────────────┘
        │                         │
        └────────────┬────────────┘
                     ▼
┌─────────────────────────────────────────────────┐
│ M05: 切片                                        │
│ · FFmpeg 按 segments 精确切片                    │
│ · 读取 system.yaml → min/max_duration 过滤      │
│ · 输出: clips/*.mp4                              │
│ · 耗时: ~1秒/片段                                │
└─────────────────────┬───────────────────────────┘
                      ▼
┌─────────────────────────────────────────────────┐
│ M06: 归档                                        │
│ · 读取 storage.path_template → 生成存储路径      │
│ · 移动文件到目标位置                              │
│ · 写入 ClipMetadata 到 SQLite                    │
│ · 触发 M12 通知                                  │
└─────────────────────────────────────────────────┘
```
